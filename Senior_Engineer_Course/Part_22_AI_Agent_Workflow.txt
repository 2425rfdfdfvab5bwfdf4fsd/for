SECTION 2: STAGE 1 — IDEA CLARIFICATION
================================================================

AI ka role: Structured brainstorming partner
Aapka role: Final vision decide karna (AI nahi karega yeh)

─── BRAINSTORMING PROMPT ───

  "I want to build: [your idea in 2-3 sentences]
  
  Help me think through this by:
  1. What clarifying questions should I answer before building?
  2. What similar products already exist? (so I know the space)
  3. What are the core technical challenges?
  4. What is a realistic MVP scope?
  5. What risks am I NOT thinking about?

  ✅ Apna core problem likho (1 sentence)

================================================================

Yeh skill aapko SENIOR engineer banati hai.
Bahut bade task = agent drift karta hai.
Bahut chhote task = efficient output.

─── THE GOLDEN RULE ───

  One task = One reviewable unit = 1-3 files max

─── DECOMPOSITION EXAMPLE ───

  Feature: "User Reviews System"
  
  BAD (one big task):

  GOOD (8 small tasks):

  Task 2: Prisma Migration
    Files: prisma/migrations/
    Do: Run migration, verify schema created correctly
    Tests: None
  
  Task 3: Review Repository
    Files: src/modules/reviews/review.repository.ts
    Do: findByProductId, findByUserId, create, delete
    Tests: src/modules/reviews/__tests__/review.repository.test.ts
  
  Task 4: Review Service — Create Review
    Files: src/modules/reviews/review.service.ts (createReview only)
    Do: Validate input, check user purchased product, create review
    Tests: src/modules/reviews/__tests__/review.service.test.ts
  
  Task 5: Review Service — Read Reviews
    Files: src/modules/reviews/review.service.ts (getProductReviews)
    Do: Paginated fetch, average rating calculation
    Tests: append to existing test file
  
  Task 6: POST /reviews endpoint
    Files: review.controller.ts, review.routes.ts, review.schema.ts
    Do: Validate, call service, return response
    Tests: review.controller.test.ts
  
  Task 7: GET /products/:id/reviews endpoint
    Files: review.controller.ts, review.routes.ts
    Do: Paginated list with filters
    Tests: append to existing test file
  
  Task 8: Integration Tests
    Files: __tests__/reviews.integration.test.ts
    Do: End-to-end tests for complete flow

─── TASK DECOMPOSITION RULES ───

  ✅ 1-3 files per task (strictly)
  ✅ Each task independently deployable / testable
  ✅ Dependencies explicit: "Task 4 depends on Task 3 being done"
  ✅ Tests included in SAME task as implementation
  ✅ Each task has clear success criteria

SECTION 10: STAGE 9 — POST-TASK REVIEW RITUAL
================================================================

Har agent task ke baad MANDATORY review karo.
Skip karna = time bomb plant karna.

─── POST-TASK REVIEW CHECKLIST ───

  STEP 1: SCOPE CHECK (30 seconds)
  □ git diff — kitni files change hui?
  □ Koi unexpected file change hui? → REJECT immediately
  □ package.json change hua? → Kyon? Approve/reject
  
  STEP 2: CODE READ (5-10 minutes)
  □ Har nai file padhni zaroori hai
  □ Business logic trace karo: happy path + 2 edge cases
  □ Error handling sab cases mein hai?
  □ TypeScript types sahi hain? 'any' use hua?
  
  STEP 3: BUILD
  □ tsc --noEmit → zero errors
  □ eslint → acceptable
  
  STEP 4: TEST
  □ npm test → all pass
  □ New tests meaningful hain? (copy-paste tests → reject)
  
  STEP 5: QUICK SECURITY SCAN
  □ Input validation hai?
  □ Auth check hai (agar required)?
  □ Sensitive data logged nahi?

─── WHEN TO REJECT ───

  REJECT if:
  ✗ Unexpected files modified
  ✗ TypeScript errors
  ✗ Tests don't pass
  ✗ Business logic wrong (trace karo)
  ✗ Pattern inconsistent with codebase
  ✗ Security hole visible
  ✗ 'any' types used
  ✗ Code duplicates existing functionality

  When rejecting:
  "The code has these specific issues:
   1. [exact issue with line number]
   2. [exact issue]
   Please fix ONLY these issues. Do not change anything else."

─── REVIEWING AI-GENERATED TESTS ───

  ❌ Tests that always pass (no assertion):

       await userService.create(data); // no expect!
     });
  
  ❌ Tests that test implementation not behavior:
     expect(userRepository.create).toHaveBeenCalledTimes(1);
     // This breaks on refactor — test behavior instead

     // Same test 5 times — doesn't add value

     it('throws 422 when user has insufficient funds', async () => {
       // Arrange
       accountRepo.findById.mockResolvedValue({ balance: 50 });
       
       // Act & Assert
       await expect(
         transferService.transfer('acc1', 'acc2', 100)
       ).rejects.toThrow(AppError);
       
       const error = await transferService.transfer(...).catch(e => e);
       expect(error.statusCode).toBe(422);
       expect(error.message).toContain('Insufficient');
     });

SECTION 14: GIVING AI AGENTS SMALL PRECISE TASKS — THE SCIENCE
================================================================

Itna important hai ke dobara cover karte hain.

─── TASK SIZE ANALYSIS ───

  VERY LARGE TASK:
  "Implement the entire authentication system"
  Problems:
  - Agent will make architectural decisions (wrong!)
  - 15+ files touched
  - Hard to review
  - Errors cascade and compound
  - Result: garbage you have to rewrite anyway
  
  LARGE TASK:
  "Implement login endpoint with JWT"
  Problems:
  - Still too vague
  - Agent decides schema, validation pattern, error format
  - 5-7 files touched
  - Harder to catch mistakes
  
  CORRECT TASK SIZE:
  "Add loginSchema (Zod) to src/modules/auth/auth.schema.ts
   following the existing registerSchema pattern (shown below).
   
   Fields: email (required email), password (required min 8 chars)
   Export as: loginSchema
   Also export: LoginInput = z.infer<typeof loginSchema>
   
   Only modify: src/modules/auth/auth.schema.ts
   Pattern to follow: [paste registerSchema]"
  
  Why this works:
  - One file
  - Exact pattern to follow
  - Clear success criteria
  - Easy to review (30 seconds)
  - If wrong: easy to fix

─── TASK BREAKDOWN FRAMEWORK ───

  When you have a big feature, break it down like this:

  Layer 1: Data (schema/migration)
    → 1-2 tasks max

  Layer 2: Data access (repository)
    → 1 task per repository
    → Include tests

  Layer 3: Business logic (service)
    → Break by METHOD — one task per method if complex
    → Always include tests

  Layer 4: HTTP layer (controller + routes + schema)
    → 1 task per endpoint or group of related endpoints

  Layer 5: Tests (integration tests)
    → 1 task for integration test file

  Each layer depends on the previous.
  Do them in order.
  Review each before moving to next.

SECTION 15: AGENT CONVERSATION MANAGEMENT
================================================================

─── LONG RUNNING AGENT SESSIONS ───

  Problem: Context window fills up → agent "forgets" earlier instructions

  Signs that agent context is degrading:
  - Agent starts making mistakes it didn't make before
  - Agent ignores your rules files
  - Inconsistent pattern usage
  - Keeps re-asking questions already answered

  Solutions:
  1. New conversation for new task (fresh context)
  2. Always paste AI_RULES.md at start of each conversation
  3. Keep conversations SHORT — one task per conversation
  4. Summarize context instead of pasting full files

─── AGENT FAILURE RECOVERY ───

  Agent ne galat kiya? Don't pile on corrections in same session.

  Instead:
  1. git checkout -- . (discard all agent changes)
  2. Start NEW conversation
  3. More specific instructions
  4. Smaller task scope
  5. Show exactly what pattern to follow (copy from codebase)

  If agent keeps making same mistake:
  → Add it explicitly to AI_RULES.md NEVER list
  → Paste the rule at start of every relevant conversation

─── ANTI-PATTERNS TO AVOID ───

     → You lose oversight, mistakes compound

  ❌ "Accept all changes without review"
     → Time bomb — will explode in production

  ❌ "Same large conversation for entire feature"
     → Context degrades, agent forgets rules
