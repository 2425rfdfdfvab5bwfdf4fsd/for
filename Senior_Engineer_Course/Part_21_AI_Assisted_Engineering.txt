SECTION 2: HOW AI CODING AGENTS WORK (Inside the Black Box)
================================================================

Ek agent ko samajhna bahut zaroori hai. Jab aap Replit Agent ya
Cursor Agent ko kuch karne ko kehte ho, andar yeh hota hai:

AGENT LOOP:

  ┌─────────────────────────────────────────┐
  │  1. OBSERVE: Codebase + Context padhna  │
  │  2. PLAN: Kya steps lene hain?          │
  │  3. ACT: File read/write, commands run  │
  │  4. OBSERVE: Naya state dekho           │
  │  5. Repeat until task complete          │
  └─────────────────────────────────────────┘

Tools jo agents use karte hain:
  - read_file (koi bhi file padh sakta hai)
  - write_file (files create/modify karta hai)
  - run_command (npm test, python script.py, etc.)
  - search_codebase (grep/search karna)
  - web_search (kuch tools mein)
  - browser (kuch tools mein)

Context window limit:
  - Agents ka context window hota hai (e.g. 200K tokens)
  - Zyada files = zyada tokens = important cheezein "forget" ho sakti hain
  - Isliye CONCISE aur FOCUSED context dena zaroori hai

Agent ki "memory" types:
  1. In-context: Current conversation + files jo usne padhe
  2. External files: ARCHITECTURE.md, AI_RULES.md, etc.
  3. Embeddings/search: Kuch agents codebase index karte hain

SECTION 3: PROMPT ENGINEERING VS AGENT INSTRUCTION ENGINEERING
================================================================

Chat prompting aur agent instructing ALAG cheezein hain.

─── CHAT PROMPT (ek sawaal, ek jawab) ───

  "How do I implement JWT refresh tokens in Node.js?"
  → AI explains, aap implement karo

─── AGENT INSTRUCTION (kaam karo, files likho) ───

  Agent ko instruction ek TASK hai, SAWAAL nahi.
  
  Structure:
    1. CONTEXT: Yeh project kya hai (ARCHITECTURE.md paste karo)
    2. TASK: Exactly kya karna hai (ek kaam, chhota)
    3. FILES: Kon si files modify karo (strict list)
    4. RULES: Kya karna FORBIDDEN hai
    5. SUCCESS: Kab task complete maana jaaye

Example — BAD Agent Instruction:
  ❌ "Make a login system"
  
  Problem:
    - Agent apna architecture invent karega
    - Naye files banayega jo project mein fit nahi karenge
    - Patterns follow nahi karega
    - Security incomplete reh sakti hai
    - Woh ek kaam mein 10 files change karega

Example — GOOD Agent Instruction:
  ✅
  "Context:
   [paste ARCHITECTURE.md — 3 lines summary]
   [paste CODE_STANDARDS.md — key rules]
   Pattern: Controller → Service → Repository (see src/modules/auth/)
   Error handling: AppError class (see src/utils/appError.ts)
   
   Task:
   Implement POST /api/v1/auth/login endpoint.
   
   Requirements:
   - Accept: { email: string, password: string }
   - Validate with Zod (see existing schema pattern in auth.schema.ts)
   - Check user exists (UserRepository.findByEmail)
   - Verify password with bcrypt.compare
   - If invalid: throw AppError(401, 'Invalid credentials')
   - If valid: return { accessToken, refreshToken, user }
   - Rate limit: already applied via middleware (do not add)
   
   Files to modify (ONLY THESE):
   - src/modules/auth/auth.schema.ts (add loginSchema)
   - src/modules/auth/auth.service.ts (add login method)
   - src/modules/auth/auth.controller.ts (add loginHandler)
   - src/modules/auth/auth.routes.ts (add POST /login route)
   
   FORBIDDEN:
   - Do NOT create new files
   - Do NOT modify any files outside src/modules/auth/
   - Do NOT add npm packages
   - Do NOT change database schema
   - Do NOT use console.log (use logger.info/error)
   - Do NOT use 'any' type
   
   Success criteria:
   - TypeScript compiles
   - POST /login with valid creds returns 200 with tokens
   - POST /login with invalid creds returns 401
   - Follows existing code style exactly"

SECTION 4: CONTEXT ENGINEERING — AGENT KA BRAIN FUEL
================================================================

Context engineering = Sahi information, sahi waqt pe, sahi format mein dena

Agent ke bina context ke behave karta hai jaise:
  "Naya intern jo pehle din aaya, koi docs nahi padhe, seedha code likh raha hai"

Agent ke saath context ke behave karta hai jaise:
  "Experienced developer jo project ko samajhta hai"

─── WHAT CONTEXT TO GIVE ───

  ZAROORI (Always include):
  □ ARCHITECTURE.md — system overview (3-5 paragraphs max)
  □ CODE_STANDARDS.md — naming conventions, patterns
  □ Relevant existing code — "yeh pattern follow karo"
  □ TypeScript interfaces/types for the feature
  □ AI_RULES.md — forbidden actions
  
  OPTIONAL (task pe depends karta hai):
  □ DATABASE.md — schema details
  □ API.md — existing endpoint patterns
  □ Error handling examples
  □ Test examples

─── CONTEXT DO'S AND DON'TS ───

  ✅ DO: Sirf RELEVANT files paste karo
  ✅ DO: Long files ko summarize karo (agent ko full file nahi chahiye)
  ✅ DO: "Follow this pattern" ke saath example paste karo
  ✅ DO: Types/interfaces hamesha include karo
  
  ❌ DON'T: Poora codebase paste mat karo
  ❌ DON'T: Unrelated modules ka context dena
  ❌ DON'T: 10 sal puraana legacy code context mein dena
  ❌ DON'T: Context as assumptions: "Agent knows our stack" — nahi jaanta

─── AI_RULES.md — Agent Constitution ───

Yeh file agent ke liye "qanoon" hai. Har project mein honi chahiye.

File: docs/AI_RULES.md

Template:
─────────────────────────────────────────────
# AI Agent Rules

## You Are
A Senior TypeScript engineer on [ProjectName].
Backend: Node.js + Express + TypeScript.
Database: PostgreSQL via Prisma ORM.
Pattern: Controller → Service → Repository.

## ALWAYS Do
- Read ARCHITECTURE.md before any change
- Follow patterns in CODE_STANDARDS.md
- Use AppError for all errors (never throw plain Error)
- Use logger (never console.log)
- Add types (never use 'any')
- Write tests for new business logic
- Use Prisma for all DB queries (never raw SQL)

## NEVER Do
- Modify files outside specified scope
- Add npm packages without explicit permission
- Change Prisma schema without being asked
- Use console.log
- Use 'any' type
- Create new architecture patterns not in docs
- Modify .env files
- Add commented-out dead code

## File Structure
src/
├── modules/          ← Feature modules only here
│   └── [feature]/
│       ├── [feature].controller.ts
│       ├── [feature].service.ts
│       ├── [feature].repository.ts
│       ├── [feature].schema.ts
│       └── [feature].routes.ts
├── utils/            ← Shared utilities
├── middleware/       ← Express middleware
└── config/           ← Config files

## Error Handling Pattern
throw new AppError(statusCode, 'message');
// NOT: throw new Error('message')
// NOT: res.status(400).json({error: '...'})

## Response Format
Always return: { data: ..., message: '...', success: true/false }
─────────────────────────────────────────────

SECTION 11: VERIFYING AI OUTPUT — THE GOLDEN CHECKLIST
================================================================

NEVER ship AI-generated code without:

  STEP 1: READ
  □ Every single line padhna — skip mat karo
  □ Business logic manually trace karo (happy path + 3 edge cases)
  □ Types check karo — 'any' use hua hai kya?
  
  STEP 2: BUILD
  □ TypeScript: tsc --noEmit — zero errors
  □ Linter: eslint — zero warnings
  
  STEP 3: TEST
  □ Existing tests pass: npm test
  □ New tests pass
  □ Manual testing: Postman ya browser se test karo
  
  STEP 4: SECURITY
  □ Input validation present? (Zod / joi)
  □ Auth/authorization check present?
  □ No SQL injection (raw queries?)
  □ Sensitive data logged to console?
  □ Secrets hardcoded?
  
  STEP 5: SCOPE
  □ git diff review karo — unexpected files changed?
  □ New npm packages added? (package.json check)
  □ DB schema changed unexpectedly?
  □ Config files touched?

─── COMMON AI MISTAKES TO CATCH ───

  ❌ Hallucinated API methods (method doesn't exist in the library)
  ❌ Wrong business logic (plausible but incorrect for YOUR domain)
  ❌ Missing error handling (happy path only)
  ❌ SQL injection via string concatenation
  ❌ Inconsistent patterns (different style from rest of codebase)
  ❌ Unnecessary complexity (over-engineered solution)
  ❌ Wrong assumptions about your requirements
  ❌ Outdated package APIs (AI trained on old data)
  ❌ Race conditions in async code
  ❌ Missing auth checks
  ❌ N+1 queries hidden in loops

SECTION 12: AGENTIC DEVELOPMENT — AUTONOMOUS AGENTS
================================================================

Agentic development = AI agents jo long multi-step tasks
autonomously complete karte hain, khud files likhte hain,
tests run karte hain, failures fix karte hain.

─── 2026 REALITY ───

  AI agents ab yeh kar sakte hain:
  ✓ Full codebase padhna
  ✓ Multiple files likhna / modify karna
  ✓ Terminal commands run karna
  ✓ Test failures dekhna aur fix karna
  ✓ Apni mistakes pe loop karna
  ✓ Web search karna

  Par ab bhi yeh nahi kar sakte:
  ✗ Business requirements independently samajhna
  ✗ Long-term architectural consequences predict karna
  ✗ Novel security threats anticipate karna
  ✗ Production impact fully judge karna

─── RISKS OF AGENTIC DEVELOPMENT ───

  1. CASCADING MISTAKES
     Agent ek galti karta hai → usi pe build karta hai → problem barhti jaati hai
     
  2. SCOPE CREEP
     "Refactor this function" → Agent poora module restructure kar deta hai
     
  3. SILENT WRONG LOGIC
     Code compile hota hai, tests pass karte hain — lekin logic galat hai
     
  4. SECURITY HOLES
     Agent authentication miss karta hai, koi rok nahi tha
     
  5. DEPENDENCY DRIFT
     Agent naye packages add karta hai jo aapne approve nahi kiye

─── SAFE AGENTIC WORKFLOW ───

  Rule 1: SMALL TASKS ONLY
    BAD:  "Build the entire payment system"
    GOOD: "Add Zod validation to POST /api/v1/orders only.
           Follow auth.schema.ts pattern.
           Only modify: src/modules/orders/order.schema.ts"
  
  Rule 2: STRICT SCOPE
    Hamesha batao: "ONLY modify these files: [list]"
    Hamesha batao: "Do NOT touch: [list]"
  
  Rule 3: REVIEW BEFORE ACCEPT
    git diff dekho — har change
    Unexpected files → reject karo
  
  Rule 4: TEST AFTER EVERY TASK
    npm test run karo
    TypeScript compile check karo
    Manual test karo
  
  Rule 5: ONE TASK, ONE REVIEW CYCLE
    Agent ne kaam kiya → aap review karo → accept/reject → next task
    Multiple tasks ek saath mat chodo
