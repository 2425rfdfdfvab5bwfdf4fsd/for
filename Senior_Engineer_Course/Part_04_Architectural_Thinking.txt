SECTION 1: ARCHITECTURE KYA HAI?
================================================================

Architecture = System ke bade structure ka design
             = Components, unke relationships, aur constraints

Analogy:
  Architecture aise hai jaise ghar ka blueprint.
  Blueprint ke bina aap seedha eenten rakhna shuru nahi karte.
  
  Software mein bhi pehle "blueprint" banao — phir code likho.

Kyun zaroori hai:
  - Bad architecture ko fix karna expensive hai
  - Good architecture future changes easy banata hai
  - Architecture security, performance, aur reliability affect karta hai

================================================================
SECTION 2: ARCHITECTURE KE FUNDAMENTALS
================================================================

--- 1. SEPARATION OF CONCERNS ---

Har component sirf ek kaam kare.

Bad Example:
  Ek function jo:
  - Database se data fetch kare
  - Business logic apply kare
  - HTML render kare
  - Email bheje
  → Yeh spaghetti code hai

Good Example:
  - DataLayer: Database se data fetch karna
  - ServiceLayer: Business logic
  - PresentationLayer: UI render karna
  - NotificationService: Emails bhejno

--- 2. MODULARITY ---

System ko independent modules mein todna jo
alag alag develop, test, aur deploy ho sakein.

Example:
  E-commerce app:
  - UserModule
  - ProductModule
  - CartModule
  - PaymentModule
  - NotificationModule
  
  Har module apna kaam kare, dusre se minimum dependency.

--- 3. COUPLING (Coupling) ---

Coupling = Kitna ek component dusre pe dependent hai

TIGHT COUPLING (bad):
  ComponentA directly ComponentB ko call karta hai
  → Agar B change ho, A bhi change karna padega

LOOSE COUPLING (good):
  Components interfaces ke zariye communicate karte hain
  → Ek change karo bina dusre ko affect kiye

--- 4. COHESION ---

Cohesion = Ek module ke andar cheezein kitni related hain

HIGH COHESION (good):
  UserService mein sirf user-related functions:
  - createUser()
  - getUser()
  - updateUser()
  - deleteUser()

LOW COHESION (bad):
  UserService mein:
  - createUser()
  - sendEmail()    ← Email service ka kaam
  - processPayment()  ← Payment service ka kaam
  - generateReport()  ← Reporting ka kaam

Rule: "High cohesion + Low coupling = Good architecture"

--- 5. ABSTRACTION ---

Implementation details chhupana, sirf interface expose karna.

Example:
  Database access:
  
  Bad (no abstraction):
    SQL query seedha controller mein likho
  
  Good (abstraction):
    UserRepository.findById(id)
    → Implementation andar chhupa hai
    → Kal PostgreSQL se MongoDB migrate karo — controller change nahi hoga

--- 6. DEPENDENCY MANAGEMENT ---

Components ke beech dependencies manage karna.

Dependency Inversion Principle:
  High-level modules low-level modules pe directly dependent nahi hone chahiye.
  Dono abstractions (interfaces) pe dependent hone chahiye.

Example:
  Bad:
    OrderService directly EmailSender class use karta hai
  
  Good:
    OrderService NotificationInterface use karta hai
    EmailSender implements NotificationInterface
    Kal SMS add karna ho → SMSSender bhi same interface implement kare

SECTION 3: ARCHITECTURE PATTERNS
================================================================

--- PATTERN 1: LAYERED ARCHITECTURE ---

Structure:
  ┌─────────────────────┐
  │  Presentation Layer  │  (UI, API endpoints)
  ├─────────────────────┤
  │  Business Logic      │  (Services, Use cases)
  ├─────────────────────┤
  │  Data Access Layer   │  (Repositories, DB queries)
  ├─────────────────────┤
  │  Database            │  (PostgreSQL, MySQL)
  └─────────────────────┘

Kab use karo:
  ✅ Simple to medium complexity apps
  ✅ Small teams
  ✅ CRUD-heavy applications
  ✅ When getting started

Kab mat use karo:
  ❌ When performance is critical (layers add overhead)
  ❌ Very complex domain logic

--- PATTERN 2: CLEAN ARCHITECTURE ---

Robert Martin (Uncle Bob) ne design kiya.

Circles (bahar se andar):
  ┌─────────────────────────────────┐
  │  Frameworks & Drivers (Express) │
  │  ┌─────────────────────────┐    │
  │  │  Interface Adapters      │    │
  │  │  ┌───────────────────┐  │    │
  │  │  │  Use Cases        │  │    │
  │  │  │  ┌─────────────┐  │  │    │
  │  │  │  │  Entities   │  │  │    │
  │  │  │  └─────────────┘  │  │    │
  │  │  └───────────────────┘  │    │
  │  └─────────────────────────┘    │
  └─────────────────────────────────┘

Rule: Dependencies sirf andar ki taraf jate hain.
      Core business logic (Entities) kisi bhi framework pe dependent nahi.

Benefits:
  - Business logic framework se independent hai
  - Test karna asaan hai
  - Framework change karo bina business logic change kiye

--- PATTERN 3: HEXAGONAL ARCHITECTURE (Ports & Adapters) ---

Core (Business Logic) ke chaaron taraf:
  - PORTS = Interfaces jo core define karta hai
  - ADAPTERS = Implementations jo ports implement karte hain

Example:
  Core business logic: "Order process karo"
  
  Ports (interfaces):
    - PaymentPort: charge(amount, customerId)
    - NotificationPort: send(userId, message)
    - OrderRepositoryPort: save(order)
  
  Adapters (implementations):
    - StripePaymentAdapter implements PaymentPort
    - EmailNotificationAdapter implements NotificationPort
    - PostgresOrderAdapter implements OrderRepositoryPort
  
  Benefit: Kal Stripe se PayPal migrate karo →
           sirf PayPalPaymentAdapter banao, core change nahi hota

--- PATTERN 4: MODULAR MONOLITH ---

Ek single deployable unit lekin internally modules mein divided.

Structure:
  app/
  ├── users/          ← User module (self-contained)
  │   ├── api/
  │   ├── services/
  │   ├── repository/
  │   └── models/
  ├── orders/         ← Order module
  ├── payments/       ← Payment module
  └── notifications/  ← Notification module

Modules communicate through well-defined interfaces,
NOT direct imports of each other's internals.

Kab use karo:
  ✅ Small to medium teams
  ✅ When microservices complexity is not justified
  ✅ Early stage products
  ✅ When bounded contexts are clear but deployment is simple

Benefits over microservices:
  - Simple deployment
  - No network overhead
  - Easier debugging
  - Less infrastructure complexity

--- PATTERN 5: MICROSERVICES ---

System ko independently deployable services mein todna.

Example: E-commerce
  - user-service (port 3001)
  - product-service (port 3002)
  - order-service (port 3003)
  - payment-service (port 3004)
  - notification-service (port 3005)

Communication:
  - HTTP/REST between services
  - Message queues (async)
  - gRPC (high performance)

Kab use karo:
  ✅ Large teams (multiple teams independently deploy)
  ✅ When services have very different scaling needs
  ✅ When technology diversity needed
  ✅ When organizational structure supports it (Conway's Law)

Kab MAT use karo:
  ❌ Small teams (operational overhead bahut zyada)
  ❌ Early stage (you don't know boundaries yet)
  ❌ When network latency is unacceptable
  ❌ When team doesn't have DevOps maturity

Microservices ke challenges:
  - Distributed system complexity
  - Network failures handle karna
  - Distributed transactions
  - Service discovery
  - Monitoring/debugging complex
  - Data consistency

--- PATTERN 6: EVENT-DRIVEN ARCHITECTURE ---

Services events ke zariye communicate karte hain (async).

Flow:
  Order placed → OrderCreatedEvent published
               → PaymentService listens → payment process karta hai
               → PaymentProcessedEvent published
               → NotificationService listens → email bhejta hai
               → InventoryService listens → stock update karta hai

Tools: Kafka, RabbitMQ, AWS SQS/SNS

Benefits:
  - Loose coupling
  - Services independently scale hote hain
  - New services easily add ho sakte hain

Challenges:
  - Event ordering
  - Duplicate events handle karna (idempotency)
  - Debugging complex (events trace karna)
  - Eventual consistency

SECTION 4: KAB KAUNSA PATTERN?
================================================================

Decision Tree:

Team size < 5?
  → Layered Architecture ya Modular Monolith

Clear domain boundaries hain?
  → Modular Monolith (pehle)
  → Microservices (jab scale/team zaroori kare)

Async processing heavy hai?
  → Event-Driven components add karo

Complex business logic?
  → Clean Architecture ya Hexagonal

High scalability needed?
  → Microservices (careful planning ke saath)

RULE: "Start with a Monolith, extract Microservices only when pain is felt"
(Martin Fowler)

SECTION 5: REAL-WORLD ARCHITECTURE EXAMPLE
================================================================

SCENARIO: E-commerce startup banana hai
Team: 3 engineers, 6 months timeline

WRONG approach:
  "Hum microservices use karein ge — scalable hoga"
  
  Result:
  - 6 services setup karna
  - Docker, Kubernetes, service mesh
  - 3 mahine infrastructure mein chali gayi
  - Product sirf 3 mahine mein bana
  - Team exhausted

RIGHT approach:
  "Pehle Modular Monolith"
  
  - Single deployable unit
  - Internal modules: users, products, orders, payments
  - Clean interfaces between modules
  - 3 mahine mein product ready
  - 3 mahine mein features aur polish
  - Jab traffic aaye aur pain points clear hon → tab extract karein

SECTION 6: DOMAIN-DRIVEN DESIGN (DDD) — BASICS
================================================================

DDD = Business domain ke hisaab se code organize karna

Key Concepts:

DOMAIN: Business area (e-commerce, healthcare, banking)

BOUNDED CONTEXT: Domain ka ek specific area
  Example:
    "User" e-commerce mein:
    - Auth context mein: credentials, sessions
    - Order context mein: shipping address, order history
    - Marketing context mein: preferences, segments

ENTITY: Unique identity wali cheez
  Example: Order (ID: 12345) — same order, even if details change

VALUE OBJECT: Identity nahi, value important
  Example: Money(100, "USD") — no ID needed

AGGREGATE: Related entities ka group (one root)
  Example: Order (root) + OrderItems + ShippingAddress

SECTION 7: ARCHITECTURE DOCUMENTATION — ADR
================================================================

ADR = Architecture Decision Record

Har bada architecture decision document karo.

Format:
  Title: [Short title]
  Status: Proposed / Accepted / Deprecated
  Context: [Problem ya situation]
  Decision: [Jo decide kiya]
  Consequences: [Positive aur negative effects]

Example ADR:

Title: Use PostgreSQL as primary database
Status: Accepted
Context: 
  Hamen ek reliable database chahiye jo complex queries,
  transactions, aur ACID compliance support kare.
  Team ko SQL aata hai.
Decision:
  PostgreSQL use karein ge primary database ke liye.
Consequences:
  Positive: ACID compliance, rich query language, proven at scale
  Negative: Horizontal scaling complex, NoSQL flexibility nahi

SECTION 8: EXERCISE
================================================================

SCENARIO:
  Ek online learning platform banana hai:
  - Students courses enroll kar sakte hain
  - Teachers content upload kar sakte hain
  - Quizzes aur assignments hain
  - Progress tracking hai
  - Payments hain
  - Video streaming hai

TASK 1: Architecture choice karo aur justify karo
  (Team: 4 engineers, 3 mahine, first version)

TASK 2: Modules identify karo

TASK 3: Ek module ka layered structure design karo

TASK 4: Ek ADR likho (database choice ke liye)

---

SAMPLE ANSWER:

Architecture Choice: Modular Monolith
Justification: Small team, tight timeline, boundaries abhi clear ho rahi hain.
               Microservices ki complexity premature hai.

Modules:
  - auth (login, signup, sessions)
  - users (students, teachers, profiles)
  - courses (content, curriculum)
  - enrollment (student-course relationship)
  - assessments (quizzes, assignments)
  - payments (subscriptions, purchases)
  - media (video upload, streaming)
  - notifications (emails, in-app)

Module Structure (courses):
  courses/
  ├── api/          ← HTTP routes
  ├── services/     ← Business logic
  ├── repository/   ← DB queries
  ├── models/       ← Data types
  └── events/       ← Events published/consumed

QUIZ — PART 4

Q1: "High cohesion, loose coupling" ka matlab kya hai?
    (a) Sab code ek jagah ho, aur sab ek dusre se connected ho
    (b) Har module apna specific kaam kare, aur modules ki dependency minimum ho
    (c) Code fast ho
    (d) Code short ho

Jawab: (b)

Q2: Microservices kab use karna NAHI chahiye?
    (a) Jab team badi ho
    (b) Jab startup early stage mein ho, team chhoti ho, aur boundaries unclear hon
    (c) Jab traffic zyada ho
    (d) Jab budget ho

Jawab: (b)

Q3: ADR kya hota hai?
    (a) Automatic Deployment Recipe
    (b) Architecture Decision Record — bade technical decisions document karne ki technique
    (c) Advanced Debugging Report
    (d) API Design Reference

Jawab: (b)

Q4: Event-driven architecture mein services kaise communicate karte hain?
    (a) Direct HTTP calls
    (b) Shared database
    (c) Events/messages ke zariye (async)
    (d) Shared memory

Jawab: (c)
