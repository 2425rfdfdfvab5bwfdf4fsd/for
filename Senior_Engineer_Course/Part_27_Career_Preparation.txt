SECTION 1: SENIOR ENGINEER INTERVIEW OVERVIEW
================================================================

Senior engineer interview usually 4-6 rounds:

1. Recruiter Screen (30 min)
   - Background, experience, motivations
   - Compensation expectations

2. Technical Screen / Coding (60 min)
   - Coding problem (DSA)
   - Sometimes take-home assignment

3. System Design (60 min)
   - Design a large-scale system
   - Architecture, trade-offs, scaling

4. Behavioral / Leadership (60 min)
   - STAR format stories
   - Leadership, conflict, failures

5. Domain / Architecture (60 min)
   - Deep dive on past project
   - Technical decisions you made

6. Team Fit / Bar Raiser
   - Culture fit
   - Senior engineers evaluate

SECTION 2: CODING INTERVIEWS
================================================================

Topics:
  - Arrays, Hash Maps, Strings (most common)
  - Trees, Graphs (medium)
  - Dynamic Programming (less frequent for senior)
  - System + API design embedded in problems

Senior coding interview difference:
  - Code quality matters (not just working solution)
  - Communication critical ("I'm thinking...")
  - Trade-offs explain karo
  - Edge cases proactively identify karo
  - Test your solution

PROCESS (follow this every time):

  1. UNDERSTAND (2-3 min):
     - Example ke through problem samjho
     - Edge cases clarify karo
     - Input/output constraints poochho
  
  2. PLAN (3-5 min):
     - Brute force approach bolo
     - Optimize karo
     - Complexity bolo
     - "Does this approach make sense before I code?"
  
  3. CODE (15-20 min):
     - Clean, readable code
     - Meaningful variable names
     - Talk while you code
  
  4. TEST (5 min):
     - Example test case
     - Edge cases (empty, one element, duplicates)
     - Trace through code

Common Senior Interview Problems:

LRU Cache:
  Design + implement
  Solution: HashMap + Doubly Linked List
  O(1) get and put

Rate Limiter:
  Design a rate limiter
  Solution: Sliding window counter with Redis

Two Sum / variants:
  Array + HashMap approach

Word Search in grid:
  DFS + backtracking

SECTION 3: SYSTEM DESIGN INTERVIEWS
================================================================

FORMAT (60 min):

  5 min: Requirements clarify karo
  10 min: High-level design
  20 min: Deep dive (specific components)
  10 min: Scale aur bottlenecks
  5 min: Trade-offs aur alternatives
  10 min: Questions

APPROACH:

Step 1: Requirements (ALWAYS clarify first)
  "Before I design, let me clarify requirements:
   - How many users? Daily active? Requests per second?
   - What are the core features? What's out of scope?
   - Read vs write ratio?
   - Consistency vs availability priority?
   - Any specific constraints?"

Step 2: Estimation (back-of-envelope)
  Show your math:
  "100 million users × 10 requests/day = 1 billion requests/day
   = 12,000 RPS average, 40,000 RPS peak
   Storage: 100M users × 1KB profile = 100GB"

Step 3: High-level design
  "I'll start with a simple design and evolve it:
   [Draw: Client → Load Balancer → App Servers → DB]"

Step 4: Component deep dive
  Pick the most interesting/challenging component
  Go deep — this is where senior candidates shine

Step 5: Scale
  "Now let's think about scaling:
   - Read replicas for DB
   - Cache layer with Redis
   - CDN for static content
   - Horizontal scaling for app tier"

Step 6: Failure modes
  "What happens when:
   - DB goes down?
   - Cache goes down?
   - One region fails?"

COMMON SYSTEM DESIGN QUESTIONS:

  Design Twitter/X
  Design Instagram
  Design YouTube
  Design Uber
  Design WhatsApp
  Design URL shortener (bit.ly)
  Design notification system
  Design rate limiter
  Design search autocomplete
  Design distributed cache

TIPS:
  - Always clarify before designing
  - Start simple, evolve
  - Justify every decision
  - Know trade-offs
  - Mention what you'd monitor

SECTION 4: BEHAVIORAL INTERVIEWS
================================================================

STAR Format:
  S — Situation (context)
  T — Task (what you needed to do)
  A — Action (what YOU specifically did)
  R — Result (outcome, measurable)

PREPARE STORIES FOR:

1. Led a complex technical project
2. Disagreed with a technical decision
3. Mentored a junior engineer
4. Dealt with a production incident
5. Made a mistake and how you handled it
6. Influenced without authority
7. Dealt with a difficult colleague
8. Made a decision with incomplete information

Example Story — Production Incident:

Situation: "Our payment service went down during Black Friday.
            Orders were failing, costing $50K/hour."

Task: "I was the senior engineer on call that night.
       I needed to identify root cause and restore service ASAP."

Action: "I immediately pulled metrics and identified DB connection
         pool exhaustion. I:
         1. First checked if users were being double-charged
         2. Identified the root cause (a slow query from new feature)
         3. Rolled back the feature flag for new checkout flow
         4. Service restored in 23 minutes
         5. Communicated status updates every 5 minutes to stakeholders
         6. Next day: Proper fix with index + postmortem"

Result: "Service restored in 23 minutes. Total revenue impact: $200K
         vs estimated $2M if issue persisted through night.
         Postmortem resulted in: better DB monitoring, index review
         process for new features, improved on-call runbooks."

SECTION 8: SALARY NEGOTIATION
================================================================

Research:

  - Glassdoor
  - Blind (anonymous)
  - LinkedIn Salary

Negotiation tips:
  1. Always negotiate — first offer rarely best
  2. Total compensation = Base + Bonus + Equity + Benefits
  3. Use competing offers (even if you prefer this company)
  4. Be specific: "Based on my research and experience, I'm targeting X"
  5. Negotiation is expected — not rude

BATNA (Best Alternative To Negotiated Agreement):
  Know your walkaway point.
  Competing offer = strong leverage.

SECTION 9: INTERVIEW MINDSET
================================================================

Nervousness ka antidote:

  Remember:
  - Interviews are conversations, not tests
  - It's mutual evaluation — you're evaluating them too
  - You don't need to know everything
  - "I don't know, but here's how I'd approach finding out" is valid
  - Ask clarifying questions — senior engineers do this

Questions TO ASK in interviews:
  "What does success look like in this role at 6 months?"
  "What's the engineering team's biggest challenge right now?"
  "How does the team make technical decisions?"
  "What's the deployment process?"
  "How does the team handle incidents?"
  "What's the on-call culture?"
  "Why is this role open?"
