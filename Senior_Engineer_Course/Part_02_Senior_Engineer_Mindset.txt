SECTION 1: CODE LIKHNE SE PEHLE KYA KARTE HAIN?
================================================================

Junior engineer:
  Problem mili → Keyboard par haath gaye → Coding shuru

Senior engineer:
  Problem mili → Ruka → Socha → Samjha → Clarify kiya
               → Todaa → Design kiya → Evaluate kiya
               → Phir implement kiya

YEH FARQ HI SAB KUCH HAI.

================================================================
SECTION 2: SENIOR ENGINEER FRAMEWORK
================================================================

Framework:
UNDERSTAND → CLARIFY → DECOMPOSE → DESIGN → EVALUATE
→ IMPLEMENT → VALIDATE → OPERATE → IMPROVE

Har step detail mein:

--- STEP 1: UNDERSTAND ---

Kya karna hai:
  - Problem ko poori tarah padhna
  - Asli problem identify karna (stated problem ≠ real problem)
  - User ka goal samajhna

Junior kya karta hai galat:
  - Pehla solution jo dikhta hai woh implement karna shuru karta hai
  - Requirements assume karta hai, poochta nahi

Real Example:
  PM ne kaha: "Search feature slow hai, speed up karo."
  
  Junior: Database query optimize karna shuru kar deta hai.
  
  Senior pehle poochta hai:
    - Kitna slow hai? (measure pehle)
    - Kaunse searches slow hain? Sab ya kuch?
    - Kitne users affected hain?
    - Yeh nayi problem hai ya pehle se thi?
  
  Result: Senior ne discover kiya ke sirf ek specific filter slow tha.
  10 minute mein fix ho gaya. Junior 2 din lagata queries optimize karne mein.

--- STEP 2: CLARIFY ---

Senior poochta hai:
  1. "Yeh feature kab tak chahiye?" (deadline)
  2. "Success kaisi dikhti hai?" (metrics)
  3. "Kitne users use karenge?" (scale)
  4. "Agar yeh kaam na kare toh kya hoga?" (criticality)
  5. "Kya pehle kuch aisa tha jo kaam nahi kiya?" (context)
  6. "Yeh feature kahan use nahi hona chahiye?" (out of scope)

EXERCISE:
  PM ne kaha: "Notifications system banana hai."
  Aap kya sawalaat puchoge? (5 minute socho)

  Senior ke sawalaat:
  - Email, SMS, push — kaunse types?
  - Real-time ya batch?
  - Kitne notifications per user per din?
  - Kya user unsubscribe kar sakta hai?
  - Priority system chahiye?
  - Notifications miss hona theek hai ya guaranteed delivery chahiye?
  - Retry logic chahiye?

--- STEP 3: DECOMPOSE ---

Bada problem chhote pieces mein todna.

Technique: "Break it down until each piece can be built independently"

Example: "E-commerce checkout banana hai"

Level 1:
  - Cart management
  - Payment processing
  - Order creation
  - Inventory update
  - Email confirmation

Level 2 (Cart management):
  - Add item to cart
  - Remove item from cart
  - Update quantity
  - Calculate total
  - Apply discount codes
  - Cart persistence (login ke baad bhi rehna chahiye)

Level 3 (Cart persistence):
  - Guest cart (localStorage?)
  - Logged-in cart (database?)
  - Cart merge (guest → logged in)
  - Cart expiry (kitni der tak?)

Yeh decomposition hi senior aur junior mein farq hai.

--- STEP 4: DESIGN ---

Implement karne se pehle design karo.

Kya design karna hai:
  - Data models
  - API contracts
  - Service boundaries
  - Error handling strategy
  - Failure modes

Tools:
  - Simple diagrams (boxes and arrows)
  - Pseudocode
  - Sequence diagrams
  - ERD (Entity Relationship Diagram)

Rule: "Design tak aane se pehle code likhna nahi."

--- STEP 5: EVALUATE ---

Apne design ko evaluate karo khud hi:

Checklist:
  □ Kya yeh scale hoga? (1000 users? 1 million users?)
  □ Kya yeh fail ho sakta hai? Kaise?
  □ Security holes toh nahi?
  □ Kya maintain karna easy hoga?
  □ Koi simpler solution hai?
  □ Kya yeh reversible hai agar galat nikla?
  □ Performance acceptable hai?
  □ Test likhna possible hai?

--- STEP 6: IMPLEMENT ---

Ab code likho — lekin:
  - Ek feature ek baar mein
  - Chhoti commits
  - Test saath mein likhna
  - Documentation update karna

--- STEP 7: VALIDATE ---

Validate karo ke jo build kiya woh sahi kaam karta hai:
  - Unit tests pass
  - Integration tests pass
  - Edge cases cover
  - Performance acceptable
  - Security check
  - Code review

--- STEP 8: OPERATE ---

Deploy ke baad kya karna hai:
  - Monitor karo (errors, latency, CPU, memory)
  - Alerts set karo
  - On-call readiness
  - Rollback plan

--- STEP 9: IMPROVE ---

Production data se seekho:
  - Users actually kaise use kar rahe hain?
  - Kya slow hai?
  - Kya confusing hai?
  - Kya improve karna chahiye?

SECTION 3: FIRST PRINCIPLES THINKING
================================================================

First Principles = Har cheez ko uske basic truth tak tod dena.

Example:
  Sawaal: "Hum notifications kaise bhejenge?"
  
  Common thinking: "SendGrid use karo, sab karte hain."
  
  First Principles:
    - Notification kya hai? → Ek message jo user tak pahuchna hai
    - User kahan hai? → Email, phone, browser
    - Message reliable hona chahiye? → Depends on type
    - Cost kya hai? → Per message
    - Agar service down ho toh? → Queue chahiye
  
  Conclusion: Simple cases mein SendGrid perfect hai.
  High volume mein apna queue system zyada cost effective.

================================================================
SECTION 4: TRADE-OFF ANALYSIS
================================================================

Engineering mein koi "perfect" solution nahi hota.
Har solution ke trade-offs hote hain.

Common Trade-offs:

1. SPEED vs ACCURACY
   - Cache use karo → Fast, lekin stale data
   - Direct DB query → Fresh data, lekin slow

2. SIMPLICITY vs FLEXIBILITY
   - Simple code → Easy to maintain, lekin hard to change
   - Flexible code → Easy to change, lekin complex

3. CONSISTENCY vs AVAILABILITY
   - Strong consistency → Data hamesha sahi, lekin slow
   - Eventual consistency → Fast, lekin kuch der ke liye stale

4. COST vs PERFORMANCE
   - Better servers → Fast, lekin expensive
   - Cheap servers → Slow, lekin cost effective

Senior Engineer ka kaam: RIGHT trade-offs choose karna given context mein.

EXERCISE: Trade-off Matrix
Problem: "Real-time chat feature banana hai"

Option A: WebSockets
  Pros: True real-time, low latency
  Cons: Complex, server resources zyada

Option B: Polling (har 2 second pe check karo)
  Pros: Simple, existing HTTP infrastructure
  Cons: Delay, extra server load

Option C: Long Polling
  Pros: Near real-time, simpler than WebSockets
  Cons: Connection management complex

Aap kya choose karoge aur kyun? (Hint: user count aur latency requirement pe depend karta hai)

SECTION 5: TECHNICAL DEBT
================================================================

Technical debt = Future mein mushkil kaam karne wala shortcut aaj karna

Types:
1. Deliberate — Consciously liya shortcut ("abhi quick fix, baad mein theek karein")
2. Accidental — Pata nahi tha ke yeh problem create karega
3. Bit rot — Purana code jo gradually outdated ho gaya

Managing Technical Debt:
  - Track karo (TODO comments, issues)
  - Prioritize karo — kaunsa debt production mein problem dega?
  - Allocate time — har sprint mein kuch time debt ke liye
  - Communicate karo stakeholders ko

Rule: "Technical debt interest ki tarah badhta hai — ignore karo toh bankrupt ho jao"

================================================================
SECTION 6: PREMATURE OPTIMIZATION SE BACHNA
================================================================

Donald Knuth ka famous quote:
"Premature optimization is the root of all evil."

Kya matlab hai:
  Jab aap optimize karte ho kisi cheez ko jab problem abhi ayi hi nahi,
  aur complicated code likhte ho jo maintain karna mushkil hai.

Example:
  Junior: "Main array use nahi karunga, linked list use karunga kyunki O(1) insertion hai"
  (Jabke array zyada cache-friendly hai aur actual data small hai)

Senior approach:
  1. Pehle simple likhao
  2. Measure karo
  3. Bottleneck dhundho
  4. PHIR optimize karo — sirf wahan jahan zaroorat hai

Rule: "Make it work → Make it right → Make it fast"

SECTION 7: BUILD vs BUY DECISIONS
================================================================

Kab khud banao? Kab existing solution use karo?

BUILD karo jab:
  - Yeh aapka core business hai
  - Off-shelf solutions fit nahi karte
  - Unique requirements hain
  - Long-term control chahiye

BUY/USE karo jab:
  - Well-tested solution exist karta hai
  - Cost kam hai
  - Maintenance burden zyada hogi
  - Time-to-market important hai

Examples:
  - Authentication → Use Auth0/Supabase (pehle se tested, secure)
  - Payment processing → Use Stripe (compliance, security)
  - Email → Use SendGrid (deliverability)
  - Apna core product → Build karo khud

================================================================
SECTION 8: REVERSIBILITY — DECISION QUALITY
================================================================

Senior engineer har decision mein sochta hai:
"Agar yeh galat nikla toh kya main wapas ja sakta hoon?"

Two-way door decisions:
  - Easy to reverse
  - Zyada risk le sakte ho
  - Fast decide karo

One-way door decisions:
  - Hard to reverse
  - Carefully socho
  - Slow decide karo

Examples:
  Two-way: Feature flag use karke feature release karna → Easily rollback
  One-way: Database schema change karna → Rollback mushkil

Amazon bezos ka rule: "Zyada tar decisions two-way doors hain — don't overthink them."

================================================================
SECTION 9: INCOMPLETE INFORMATION MEIN DECISIONS LENA
================================================================

Aapke paas kabhi poori information nahi hogi.
Senior engineer phir bhi decisions leta hai.

Framework:
1. Jo pata hai woh list karo
2. Jo nahi pata woh list karo
3. Assumptions clearly state karo
4. Reversible path choose karo jab possible
5. Decision log karo

Example:
  "Hum nahi jaante ke traffic kitna hoga launch pe"
  
  Junior: Wait karta hai jab tak pata chale
  Senior: 
    - Assumption: 10,000 DAU first month
    - Monitoring set karo
    - Auto-scaling configure karo
    - Rollback plan rakho
    - Launch karo, data dekho, iterate karo

SECTION 10: COMMON JUNIOR MISTAKES (JO SENIOR NAHI KARTA)
================================================================

1. ❌ Requirements assume karna → ✅ Pehle clarify karo
2. ❌ Seedha code likhna → ✅ Pehle design karo
3. ❌ Happy path sirf test karna → ✅ Edge cases aur failure modes test karo
4. ❌ "Mere machine pe kaam karta hai" → ✅ Production environment socho
5. ❌ Over-engineer karna → ✅ Simple raho, baad mein iterate karo
6. ❌ Performance baad mein sochna → ✅ Basic perf requirements pehle define karo
7. ❌ Security ignore karna → ✅ Security by design
8. ❌ Documentation nahi likhna → ✅ Code likhte waqt doc likho
9. ❌ Monitoring add nahi karna → ✅ Observability deploy ke saath
10. ❌ Akele kaam karna → ✅ Collaborate, communicate, review lo
