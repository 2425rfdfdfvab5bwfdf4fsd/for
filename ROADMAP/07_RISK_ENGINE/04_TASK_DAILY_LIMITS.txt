OBJECTIVE:
    Implement DailyLimitsChecker that tracks daily trade count and daily P&L
    and blocks new trades when limits are reached.

CONTEXT:
    Daily limits:
    - MAX_DAILY_TRADES: 3 (configurable)
    - MAX_DAILY_LOSS_PERCENT: 2.0% (configurable, calculated on equity)
    - Limits reset at the start of each new BROKER trading day (broker server midnight,
      NOT UTC midnight — see DAILY_RESET_ALGORITHM below and CHG-008)

    When daily loss limit is hit → ALL trading halted for the day.
    When trade count limit is hit → No new entries (management of existing
    trades continues normally).

DEPENDENCIES:
    - Phase 04 (Data Layer): daily trade records from database
    - Phase 03 (MT5): current equity

================================================================================
DAILY_RESET_ALGORITHM — MANDATORY (CHG-003, CRITICAL SAFETY)
================================================================================

Without this, a restarted bot could bypass the 2% daily loss limit entirely.

DAY-START EQUITY RECORDING:
  At the FIRST market tick/scan of each broker trading day:
    1. Query daily_stats table for today's date (using broker server date)
    2. If NO record exists (new day): INSERT daily_stats record with:
         date          = today_broker_date
         day_start_equity = current_equity (from MT5 account_info)
         trades_count  = 0
         realized_pnl  = 0.0
    3. If record EXISTS: use existing day_start_equity (do NOT overwrite)

DAILY LOSS CALCULATION:
  daily_loss_pct = (day_start_equity - current_equity_including_floating) /
                   day_start_equity * 100
  current_equity_including_floating = account_info.equity  (NOT balance)
  This includes floating P&L of open positions — prevents hiding losses.

BROKER DAY BOUNDARY:
  Use SERVER_UTC_OFFSET_HOURS from config (default: +2, most MT5 brokers).
  Broker date = UTC datetime + SERVER_UTC_OFFSET_HOURS
  Do NOT use system clock or UTC midnight as the day boundary.
  See CHG-008 for server timezone detection.

ON BOT RESTART — STATE RECOVERY PROCEDURE:
  1. Query daily_stats for today's broker date
  2. If record FOUND: use stored day_start_equity and trades_count
  3. If record MISSING (e.g., DB corruption or first run):
     - Use current equity as day_start_equity (CONSERVATIVE FALLBACK)
     - Log WARNING: "Daily state missing on restart — using current equity
                     as day_start. Loss limit may be less restrictive today."
     - Set trades_count = 0 (cannot reconstruct without DB record)
  4. Log: "Recovered daily state: start_equity={}, trades={}, loss_pct={}%"

DB SCHEMA REQUIRED IN PHASE 04:
  daily_stats table must have columns:
    date TEXT (PRIMARY KEY — broker date string YYYY-MM-DD)
    day_start_equity REAL
    trades_count INTEGER
    realized_pnl_today REAL

================================================================================
DETAILED REQUIREMENTS:
    class DailyLimitsChecker:
        check(daily_stats: DailyStats, current_equity: float) -> LimitCheckResult

    DailyStats (from database):
        trades_today: int
        starting_equity: float
        realized_pnl_today: float

    LimitCheckResult:
        allowed: bool
        reason: str | None    # "DAILY_LOSS_LIMIT", "DAILY_TRADE_LIMIT", None

    Daily loss calculation:
        loss_pct = (starting_equity - current_equity) / starting_equity * 100
        if loss_pct >= MAX_DAILY_LOSS_PERCENT: BLOCKED

             {date: today_broker, day_start_equity: 10000.0, trades_count: 1}

          6. Assert result.reason == "DAILY_LOSS_LIMIT"
          7. Assert fresh_checker reads day_start_equity=10000.0 from DB
             (NOT current equity — which would reset the loss tracking)
